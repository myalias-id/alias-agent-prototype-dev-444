Executive Summary

This report covers a full inspection of the alias-agent-prototype Next.js application — a real-time AI avatar chat system with VRM 3D models, WebSocket streaming, audio processing, and blockchain integrations. Every source file was examined: API routes, Zustand stores, React components, hooks, utility libraries, and configuration files.

The codebase is well-architected overall, with good use of TypeScript, Zustand, and modular separation. However, 3 critical security vulnerabilities were identified (including admin secrets leaking to the browser), along with pervasive code duplication, missing tests, inconsistent API response shapes, memory leaks, and a number of React best-practice violations.

The findings are grouped into 12 categories, ordered by severity. Concrete fix examples are provided for the most impactful issues.

1. Security Vulnerabilities

Critical Admin Secret Exposed in Client-Side Code

Three client-side files reference process.env.BACKEND_API_ADMIN_SECRET directly in fetch() Authorization headers. In Next.js, only NEXT_PUBLIC_* environment variables are embedded into the browser bundle. All others are undefined at runtime — meaning these calls silently send Authorization: Bearer undefined. If anyone ever adds the NEXT_PUBLIC_ prefix to work around this (a common mistake), the admin secret would be publicly visible in every browser's network tab.
Files: src/hooks/useVrmInitializer.ts · src/hooks/useBackgroundInitializer.ts · src/components/vrm/VrmManager.ts 
// ❌ Current — client component using a server-only secret
const response = await fetch('/api/agent/ms-vrm', {
  headers: {
    Authorization: `Bearer ${process.env.BACKEND_API_ADMIN_SECRET}`, // undefined in browser
  },
});
Fix: The client should call the Next.js API route without any Authorization header. The API route (running server-side) then attaches the admin secret before proxying to the backend. This is already done correctly in the API routes — the client-side hooks should simply drop the Authorization header entirely. 
// ✅ Fixed — client calls its own API route without a secret
const response = await fetch('/api/agent/ms-vrm');  // No auth header needed

Critical Hardcoded Empty Token in getMetadata.ts

The getMetadata server function calls the backend with a literal empty string inside the Bearer token: Authorization: `Bearer ''`. This is a hardcoded malformed credential. If the backend ever starts validating this header, it will silently break server-side metadata generation (SSR/SEO). More importantly, it signals that authentication intent here is unclear and was patched over.
File: src/lib/getMetadata.ts · Line 16
// ❌ Current
Authorization: `Bearer ''`,
Fix: Either use the proper server-side secret or remove the Authorization header if this endpoint is public. 
// ✅ Option A: use the actual secret (server-side only — safe)
Authorization: `Bearer ${process.env.BACKEND_API_ADMIN_SECRET}`,

// ✅ Option B: if the /agent/:id endpoint is public, omit the header
// (no Authorization header)

Critical Overly Permissive CORS & Framing Headers

The Next.js config sets Access-Control-Allow-Origin: * for all API routes and X-Frame-Options: ALLOWALL on all paths globally. This allows any website to:
  • Embed the app in an iframe (clickjacking attack surface)
  • Make cross-origin requests to all API endpoints (CSRF risk without additional token-based protections)
The embed use-case is valid, but these headers should be scoped to the /embed path only.
File: next.config.js · headers() function
// ❌ Current — applied globally
{
  source: '/:path*',
  headers: [{ key: 'X-Frame-Options', value: 'ALLOWALL' }],
},
Fix: Restrict permissive headers to the /embed route only. For API routes, specify the allowed origins explicitly or use CSRF tokens. 
// ✅ Fixed — only the embed page is frameable
{
  source: '/:path*',
  headers: [{ key: 'X-Frame-Options', value: 'SAMEORIGIN' }],
},
{
  source: '/embed/:path*',
  headers: [{ key: 'X-Frame-Options', value: 'ALLOWALL' }],
},

High Google Analytics ID Hardcoded in layout.tsx

The GA measurement ID G-QDK3X6G6EF is hardcoded directly in RootLayout. Hardcoding analytics IDs prevents easy rotation, environment-specific tracking (dev vs prod), and complicates open-sourcing the project.
File: src/app/layout.tsx · Line 68
// ❌ Current

Fix: Move to an environment variable.

High Missing Input Validation in chat-os/generate Route

The /api/chat-os/generate endpoint accepts an array of messages and forwards them directly to OpenRouter without validating: the number of messages, the size of each message, or the content type. A malicious actor could send an enormous payload (thousands of messages, megabytes of text) to exhaust API credits or cause downstream rate-limit bans.
File: src/app/api/chat-os/generate/route.ts
Fix: Add message count and per-message length limits. 
const MAX_MESSAGES = 50;
const MAX_MESSAGE_CHARS = 4000;

if (messages.length > MAX_MESSAGES) {
  return errorResponse('Too many messages', 400);
}
const valid = messages.every(
  (m: unknown) =>
    typeof m === 'object' &&
    m !== null &&
    typeof (m as { content: unknown }).content === 'string' &&
    (m as { content: string }).content.length <= MAX_MESSAGE_CHARS
);
if (!valid) return errorResponse('Invalid message format or size', 400);

2. Code Duplication

High AgentManager Component Duplicates useAgentInitializer Hook

src/components/agent/AgentManager.ts and src/hooks/useAgentInitializer.ts contain line-for-line identical logic: both read NEXT_PUBLIC_AGENT_ID, parse the agentId query param, call fetchAgentById, set VRM, and set background. context-wrapper.tsx uses the hook; AgentManager is also imported (used elsewhere). This creates a maintenance hazard — any change must be made in two places.
Files: src/components/agent/AgentManager.ts · src/hooks/useAgentInitializer.ts
Fix: Delete AgentManager.ts and have all consumers use useAgentInitializer. The hook is already the canonical version used by ContextWrapper.

High useVrmInitializer Duplicates VrmManager

The same duplication exists for VRM loading. src/hooks/useVrmInitializer.ts and src/components/vrm/VrmManager.ts share almost identical fetchVrmsAndAnimations logic, including the same fallback local VRM object (Lucy). Additionally, both files define the getVRMBlendShapeType / GetVRMBlendShapeType function (identical algorithm, different capitalisation).
Files: src/hooks/useVrmInitializer.ts · src/components/vrm/VrmManager.ts
Fix: Consolidate into the hook. Remove VrmManager.ts or reduce it to a thin wrapper. Extract getVRMBlendShapeType to a shared utility (e.g., src/lib/utils/vrm-utils.ts).

Medium Utility Functions Duplicated Across Modules

Three utility functions are independently re-implemented in both src/lib/utils.ts and src/lib/utils/chat-utils.ts:

Medium JSON_HEADERS Constant Duplicated in API Routes

The constant const JSON_HEADERS = {'{ \'Content-Type\': \'application/json\' }'} is independently declared in at least 3 API route files: ms-agent/route.ts, places/route.ts, and transcribe/route.ts.

3. TypeScript Strictness

High no-unused-vars Rule Disabled Globally

The ESLint config sets '@typescript-eslint/no-unused-vars': 'off'. This means unused variables, dead imports, and stale function arguments are silently ignored across the entire codebase, making it hard to identify dead code and increasing bundle size.
File: .eslintrc.js · Line 46
Fix: Enable the rule with practical options to reduce noise. 
// In .eslintrc.js
'@typescript-eslint/no-unused-vars': ['warn', {
  argsIgnorePattern: '^_',
  varsIgnorePattern: '^_',
}],

High Pervasive use of any with ESLint Suppression

Multiple files disable the @typescript-eslint/no-explicit-any rule at the file level via /* eslint-disable @typescript-eslint/no-explicit-any */. Using any defeats the purpose of TypeScript and can mask real type errors.
Files: src/components/vrm/Model.ts · src/components/vrm/VrmManager.ts · src/hooks/useAudioRecording.ts · src/components/common/audio-visualizer-6.tsx · src/lib/utils/chat-utils.ts
Fix: Replace any with proper types. For browser vendor APIs, use the existing src/types/browser.d.ts or specific narrowing. For RecordRTC, use its published types or create a minimal interface. Remove the top-level eslint-disable comments and suppress inline only where genuinely unavoidable.

Medium Agent Store Initialised with null for Non-Nullable Type

In useAgentStore.ts, the store is initialised with agent: null, but the interface declares agent: IAgent (not IAgent | null). This type mismatch is silently ignored and can cause unexpected null-access runtime errors throughout the app.
File: src/store/useAgentStore.ts · Lines 90, 100
// ❌ Current interface vs initialisation mismatch
interface AgentStoreState {
  agent: IAgent;            // non-nullable
  ...
}
create<AgentStoreState>(... {
  agent: null,              // null!
  ...
})
Fix: Change the interface to agent: IAgent | null and add null checks at call sites.

4. Error Handling Inconsistencies

High Inconsistent API Response Shapes

API routes return different success/error shapes with no common contract:
RouteSuccess ShapeError Shape
ms-agent GET{"{ status: 'success', data }"}{"{ status: 'fail', message }"}
transcribe POST{"{ status: 'ok', text }"}{"{ status: 'fail', message }"}
chat-os/generate{"{ success: true, choices }"}{"{ error, details }"}
places GET{"{ status: 'success', data }"}{"{ status: 'fail', message }"}
ms-vrm GET{"{ status: 'success', data }"}{"{ error }"}
Fix: Standardise on a single response envelope. The errorResponse / successResponse helpers in src/lib/api-response.ts are a good start — use them everywhere. Update transcribe to use status: 'success' and chat-os/generate to match the same envelope.

Medium Mixing axios and fetch in the Same File

ms-agent/route.ts uses the native fetch API for the GET handler and axios for the POST handler — in the same file with no particular reason. This adds bundle weight (axios is ~14 KB gzipped) and inconsistency.
File: src/app/api/agent/ms-agent/route.ts
Fix: Standardise on native fetch for all Next.js API routes (edge runtime already requires it). Remove the axios import from this file.

Medium Silent Failure Pattern in fetchAgentById

fetchAgentById in the agent store catches all errors and returns undefined silently. Callers that await it have no way to distinguish between "agent loaded", "agent not found", and "network error", leading to subtle UI bugs where a loading spinner spins forever.
File: src/store/useAgentStore.ts · fetchAgentById
Fix: Return a typed result or throw, and set an error state in the store. The setIsLoading(false) call should be moved to a finally block so it always executes.

5. Performance Issues

High WebSocket Store Persists Non-Serializable Objects

useSocketChatStore is wrapped with Zustand's persist middleware. The store state contains socket: WebSocket | null, visualizerAnalyser: AnalyserNode | null, and visualizerGainNode: GainNode | null — all non-serializable browser objects. Attempting to serialize these to localStorage will silently fail and may corrupt persisted state or throw hard-to-diagnose errors.
File: src/store/useSocketChatStore.ts · persist() wrapper
Fix: Use Zustand's partialize option to only persist serializable fields. 
persist(
  (set, get) => ({ ... }),
  {
    name: 'socket-chat-store',
    partialize: (state) => ({
      userId:    state.userId,
      sessionId: state.sessionId,
      volume:    state.volume,
      messages:  state.messages,
    }),
  }
)

High Unbounded Model Cache in VrmComponent

modelCacheSignal in VrmComponent.tsx is a module-level Preact signal that caches loaded VRM models by URL with no eviction policy. Each loaded VRM can be 20–100 MB in memory. Over a long session with multiple agents, this will exhaust the browser's memory budget and cause page crashes, especially on mobile.
File: src/components/vrm/VrmComponent.tsx · modelCacheSignal
Fix: Implement an LRU cache with a maximum of 2–3 entries (enough for the current + previous VRM). Dispose of evicted VRM resources using the existing disposeOfModel utility in src/lib/utils/garbage-collection.ts.

Medium Unthrottled Resize Listener in agent-chat.tsx

A useLayoutEffect in agent-chat.tsx attaches a resize event listener that calls setContainerWidth synchronously on every resize event. Without debouncing, this fires hundreds of times per second during a resize drag, triggering a full component re-render each time.
File: src/components/agent/agent-chat.tsx · useLayoutEffect
Fix: Debounce the resize handler or use the existing debounce utility from src/lib/utils.ts. Alternatively, replace with a ResizeObserver (already a browser API) which is more efficient than window resize events. 
const updateWidth = debounce(() => {
  if (containerRef.current) {
    setContainerWidth(containerRef.current.offsetWidth);
  }
}, 100);
window.addEventListener('resize', updateWidth);

Medium Production console.log Statements in Hot Paths

Model.ts has multiple console.log calls inside animation transition logic that runs every animation frame (via requestAnimationFrame). Logging on every frame causes significant garbage collection pressure and can noticeably slow down rendering. Examples: [Model] Starting gradual transition, [Model] Gradual transition completed.
File: src/components/vrm/Model.ts · startGradualTransition
Fix: Remove or gate all animation-path logs behind a debug flag: 
const DEBUG = process.env.NODE_ENV === 'development' && process.env.NEXT_PUBLIC_DEBUG_ANIM;
if (DEBUG) console.log('[Model] Starting gradual transition:', ...);

Medium Double-Fetch Risk: useVrmInitializer + VrmManager Both Active

ContextWrapper calls useVrmInitializer (which fetches VRMs from the API). VrmManager is a separate component with identical fetch logic. If both are mounted simultaneously (different pages or layout levels), the API will be called twice, wasting bandwidth and potentially causing state race conditions.
Fix: Consolidate to one data-fetching source (the hook in ContextWrapper) and delete VrmManager.ts as a standalone component.

6. Memory Leaks

High Module-Level Mutable State in useSocketChatStore

Four mutable variables are declared at module level outside the Zustand store in useSocketChatStore.ts: 
let healthCheckInterval: ReturnType<typeof setInterval> | null = null;
let audioBuffer: ArrayBuffer | null = null;
let textContent = '';
let audioGeneration = 0;
These variables: (1) are never reset when the store is rehydrated or when tests create a fresh store instance; (2) are invisible to Zustand DevTools; (3) can hold onto large ArrayBuffer objects even after the conversation ends; (4) make the store impossible to unit-test reliably. The healthCheckInterval reference is particularly dangerous — if connect() is called multiple times rapidly, there is a window where the old interval is not cleared before the new one is set.
Fix: Move all of these into the Zustand store state. Use partialize to exclude audioBuffer from persistence (it's binary data). The audioGeneration counter is already a clean pattern — keep it but document it.

Medium Debug Intervals Not Cleaned Up in Model.ts

Model.ts has private fields _debugInterval and _lipSyncDebugInterval. The startDebugLogging and enableLipSyncDebug methods set these intervals, but they are commented out in the constructor. If they are ever uncommented or called programmatically, there is no guarantee the intervals are cleared when the VRM model is unloaded (unLoadVrm).
File: src/components/vrm/Model.ts
Fix: Either remove the debug infrastructure entirely, or ensure unLoadVrm() clears both intervals before disposal.

Medium AudioContext Not Closed on Component Unmount

getAudioContext() in src/lib/audioContext.ts returns a module-level singleton. If the React component tree is unmounted (e.g., during hot module reload or page navigation), the AudioContext is never closed. Browsers limit the number of simultaneously open AudioContext instances (typically 6), which can cause audio failures on subsequent mounts.
Fix: Expose a closeAudioContext() function and call it in the appropriate cleanup. Alternatively, use a useEffect cleanup hook that calls ctx.close().

7. Architecture & Design

High useSocketChatStore Violates Single Responsibility

useSocketChatStore.ts (~600+ lines) is a monolithic Zustand store that manages: WebSocket lifecycle, message streaming, audio chunk queuing, audio playback (Web Audio API), VRM lip-sync coordination, animation emotion changes, visualizer nodes, volume control, health-check pings, session persistence, and reconnection logic. This violates the Single Responsibility Principle and makes the file extremely hard to maintain, test, or reason about.
Fix (incremental): Extract concerns into separate stores or services:
  • useAudioPlaybackStore — queue, play, volume, Web Audio API nodes
  • useWebSocketStore — connection lifecycle, reconnect, health-check
  • useChatMessagesStore — messages array, current streaming message

Medium localStorage Accessed Directly Inside Zustand Actions

The WebSocket store directly reads and writes localStorage in multiple places (localStorage.setItem('session_id', ...), localStorage.getItem('session_id')). This couples the store to the browser environment, makes server-side usage impossible, and makes testing require localStorage mocks.
Fix: Abstract localStorage behind a thin service: 
// src/lib/services/session-storage.service.ts
export const sessionStorage = {
  getSessionId: () => localStorage.getItem('session_id'),
  setSessionId: (id: string) => localStorage.setItem('session_id', id),
};
Or use Zustand's built-in persist middleware with a localStorage adapter (already partially done — complete the migration).

Medium Deprecated audioConstants.ts Still Imported

src/lib/audioConstants.ts is explicitly marked @deprecated and re-exports from src/lib/constants/audio.ts. However, if any file still imports from the old path, it carries the overhead of an indirection layer. Deprecated files that are not actively being removed create confusion about which import path is correct.
Fix: Search for all imports of audioConstants and migrate them to @/lib/constants. Once all references are updated, delete audioConstants.ts.

8. React Best Practices

High Missing useEffect Dependencies

Several useEffect hooks specify incomplete dependency arrays, which means they can run with stale closures or fail to re-run when they should:
FileIssue
useAgentInitializer.ts[fetchUpdateTrigger] — missing fetchAgentById, setIsLoading
AgentManager.tsSame issue
useBackgroundInitializer.ts[fetchUpdateTrigger] — missing store setters
useVrmInitializer.ts[fetchUpdateTrigger] — missing store functions
Fix: Either add all referenced values to the dependency array, or wrap store functions with useCallback (Zustand already returns stable refs — just add them). Alternatively, use the eslint-plugin-react-hooks exhaustive-deps rule to auto-detect these.

Medium useVRMStore.getState() Called Outside React Hook Context

In VrmComponent.tsx, useVRMStore.getState() is called at component render time (not inside a useEffect or event handler): 
const {'{ playIdleAnimationById, animationDictionary }'} = useVRMStore.getState();
This bypasses React's reactive subscription system. If the store updates, the component will not re-render to pick up the new values — it will continue using the stale references captured at first render.
File: src/components/vrm/VrmComponent.tsx
Fix: Use the hook form useVRMStore((state) => state.playIdleAnimationById) so the component subscribes to changes properly.

Medium agent-chat.tsx Has Excessive State & Re-render Risk

agent-chat.tsx maintains 12 separate useState calls and subscribes to 4 different Zustand stores. Several derived boolean values (shouldShowVRM, shouldShowVisualizer, etc.) are computed inline on every render without useMemo, and a containerWidth state update on resize can trigger a re-render of the entire 400-line component tree.
Fix:
  • Wrap derived booleans in useMemo
  • Split the component into smaller sub-components (e.g., AvatarSection, ChatSection)
  • Use React.memo on pure sub-components to reduce re-render propagation

9. API Design Consistency

Medium Unused Parameter in ms-vrm Route

The GET handler in ms-vrm/route.ts declares request: Request as a parameter but never reads it — no query params, headers, or body are accessed.
File: src/app/api/agent/ms-vrm/route.ts
// ❌ Current — unused parameter
export async function GET(request: Request) {
Fix: Remove the unused parameter: export async function GET() {'{}'}

Medium ms-agent Route Missing Request Rate Limiting

The /api/agent/ms-agent GET endpoint fetches all agents from the backend and is publicly accessible. Without any rate limiting, it can be abused to scrape agent data or cause excessive load on the backend. The chat-os/generate endpoint forwards to an expensive LLM API with no rate limiting either.
Fix: Implement rate limiting using the Cloudflare Workers rate limiting API (the app uses @opennextjs/cloudflare) or an edge middleware approach. At minimum, add a Cache-Control header to the agents list endpoint so CDN caches can absorb repeated reads.

Low Places Route Interface Defined in Route File

The PlaceResult interface is exported from the API route file (places/route.ts). API route files should not export business-logic types — this creates an unusual import path and couples type definitions to the routing layer.
Fix: Move PlaceResult to src/types/common.ts or a new src/types/places.ts.

The edge runtime already has native fetch available. Removing axios would reduce the server-side bundle and eliminate one dependency to maintain.

Low Next.js 14 — Consider Upgrading to Next.js 15

The project uses next@14.2.26. Next.js 15 brings significant improvements: faster Turbopack dev builds, improved React Server Components, better caching defaults (opt-in instead of default), and important security patches. Given the project already uses @opennextjs/cloudflare which supports Next.js 15, the upgrade path is well-supported.
Fix: Run npx @next/codemod@latest upgrade to get automated migration assistance. Test thoroughly, especially the Suspense boundaries and API route caching behaviour changes.

12. Testing

High Zero Tests in the Codebase

No test files (*.test.ts, *.test.tsx, *.spec.ts, *.spec.tsx) were found anywhere in the project. There is no testing framework configured (no Jest, Vitest, Playwright, or Cypress setup). This means:
  • Regressions in critical paths (WebSocket handling, audio playback, VRM animation) go undetected until production
  • Refactoring is high-risk with no safety net
  • The complex parsing logic in chat-utils.ts and parse-response.ts is unverified
  • API route validation logic cannot be tested without spinning up the full server
Fix (prioritised by ROI):
  1. Unit tests for pure utilities — parseSocketResponse, mapFacialEmotionToVRM, parseMessageContent, extractGoogleMapsPlaceName. These are pure functions with no dependencies. Use Vitest (zero config with Next.js).
  2. API route tests — Test input validation, error shapes, and auth header logic. Use next-test-api-route-handler or the new experimental_msw support.
  3. Integration tests for stores — Test Zustand store actions (connect, disconnect, sendMessage) with a mocked WebSocket.
  4. E2E smoke tests — Use Playwright to verify the chat flow loads and can send a message.
